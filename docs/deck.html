No. 04第 i 章 · 是什么
67子代理

内置 67 个专用子代理

planner、architect、tdd-guide、code-reviewer、security-reviewer、build-error-resolver、doc-updater 等。主代理把长任务委派给专家，避免单 Agent 跑偏，显著提升复杂任务的完成率。另有 271 个 skills、92 个 commands 在背后支撑。

No. 05第 i 章 · 是什么

五大基石

Agent · Skill · Command
Rule · Hook

Agent & Skill — 专家 + 工作流

子代理（Agent）是限定职责的专属 AI 角色；技能（Skill）是 271 个可复用工作流。两者配合让长任务被专业角色按标准流程完成。

Command & Rule — 入口 + 约束

命令（/plan、/tdd 等）是工作流入口；规则（Rule）是始终生效的编码与安全约束，例如禁提交 console.log、强制类型检查。

Hook & AgentShield — 自动化防御

钩子（Hook）在工具调用前后、会话开始结束自动触发；AgentShield 做静态与动态安全审查，拦截配置漏洞与指令注入。

speaker notes五大基石分三层讲：Agent+Skill 是"做事的人"，Command+Rule 是"入口和规则"，Hook+AgentShield 是"自动防御"。强调它们联动：Rule 不是文档，是 AI 始终遵守的硬约束；Hook 不是脚本，是事件驱动的自动化层。

命令 —
6 个日常入口

92 个命令文件、59 个全局 slash commands 中，这 6 个是日常开发最高频的工作流入口。

speaker notes不要被 92 这个数字吓到，绝大多数开发者日常只需要这 6 个。它们各自对应一个完整工作流（不是单次动作）：/plan 背后是 planner agent + 风险识别，/verify 背后是 build/lint/test/type-check 全链路。

No. 07第 ii 章 · 命令

最常用的一组

何时用哪个命令

/plan 先澄清需求、识别风险、生成实施计划 需求不清 · 任务较大
/tdd 先写失败测试，再实现，再验证覆盖率 测试优先推进
/code-review 检查代码质量、安全、可维护性 代码写完后
/build-fix 自动识别语言并修复常见构建问题 构建 / 类型 / 编译挂了
/verify 跑完整验证循环：build、lint、test、type-check 改动准备收尾
/quality-gate 按项目标准做提交前质量闸门检查 提交前最后一道闸
speaker notes6 个命令按"开发时序"排：先 /plan 想清楚，再 /tdd 写代码，/build-fix 救火，/code-review 审查，/verify 全链路验证，最后 /quality-gate 把关提交。这一张是速查表，建议观众拍照保存。

No. 09第 iii 章 · 工作流

典型 ECC 协作链

一条端到端的任务流水线

阶段 i

规划

planner 澄清需求、识别风险，architect 划边界、给权衡。先想清楚再动手。

阶段 ii

开发

tdd-guide 强制测试优先，build-error-resolver 接管类型 / 编译错误，让循环不停摆。

阶段 iii

审查

code-reviewer 查质量与可维护性，security-reviewer 扫漏洞与配置风险，双轨并行。

阶段 iv

验证

/verify 跑 build / lint / test / type-check 全链路，/quality-gate 把提交前的最后一道闸。

speaker notes四个阶段对应四组子 Agent：规划 / 开发 / 审查 / 验证。重点不是记住每个 agent 名字，而是理解"长任务被拆成可编排的链路"——这是 ECC 区别于原生 Claude Code 的核心增量。调试场景下 build-error-resolver 特别有用。
